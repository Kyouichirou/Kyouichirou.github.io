2.1 Loguru多进程支持

关于在 Windows 上的多进程支持, 文档是如此描述的: 事情在Windows上有点复杂.

Things get a little more complicated on Windows. Indeed, this operating system does not support forking, so Python has to use an alternative method to create sub-processes called " spawning" . This procedure requires the whole file where the child process is created to be reloaded from scratch. This does not interoperate very well with Loguru, causing handlers to be added twice without any synchronization or, on the contrary, not being added at all (depending on add() and remove() being called inside or outside the __main__ branch). For this reason, the logger object need to be explicitly passed as an initializer argument of your child process:

2.2 可行方案

首先尝试修改上述的代码.

其中:

def complete(self) -> AwaitableCompleter: ...

logger.complete()

这是个可当等待的函数.

import multiprocessing
import time
from loguru import logger

logger.remove()
logger.add(
    "process.log",
    level="INFO",
    enqueue=True,  # 关键: 通过队列实现多进程安全写入
    format="{time} | 进程 {process.id} | {message}"  # 显示进程ID
)
# 进程任务: 记录日志
def process_task(task_id):
    logger.info(f"任务 {task_id} 第 0 次执行")
    time.sleep(0.1)  # 模拟耗时操作

async def test_a():
    await logger.complete()

if __name__ == "__main__":

    processes = []
    for i in range(2):
        p = multiprocessing.Process(
            target=process_task,
            args=(i,)
        )
        processes.append(p)
        p.start()

    # 等待所有进程结束
    for p in processes:
        p.join()
    import asyncio
    # 确保队列中所有日志都被处理
    asyncio.run(test_a())

import multiprocessing
import time
from loguru import logger

logger.remove()
logger.add(
    "process_pool.log",
    level="INFO",
    enqueue=True,  # 关键: 通过队列实现多进程安全写入
    format="{time} | 进程 {process.id} | {message}"  # 显示进程ID
)

# 进程任务: 记录日志
def process_task(task_id):
    for k in range(3):
        logger.info(f"任务 {task_id} 第 {k + 1} 次执行")
        time.sleep(0.1)
    return task_id

if __name__ == "__main__":
    with multiprocessing.Pool(1) as pool:
        arr = (pool.apply_async(process_task, (task_id,)) for task_id in range(2))
        for task in arr:
            print(task.get())

再修改成用进程池来管理执行, 可以正常打印所有日志.

import multiprocessing
from loguru import logger
import time

def process_task(_logger, task_id):
    for k in range(3):
        _logger.info(f"任务 {task_id} 第 {k + 1} 次执行")
        time.sleep(0.1)
    return task_id

if __name__ == "__main__":
    logger.remove()  # Default "sys.stderr" sink is not picklable
    logger.add("test_file.log", enqueue=True)

    processes = []
    for i in range(2):
        p = multiprocessing.Process(
            target=process_task,
            args=(logger, i,)
        )
        processes.append(p)
        p.start()

    for p in processes:
        p.join()

正常打印所有的日志, 这种方式是文档中提及的在Windows下多进程的使用示例之一, 其关键在于将logger示例作为参数传递到每个子进程中.

上述方法也是最为直观, 容易理解的, 推荐使用.

首先来看这部分

在继续之前, 需要来了解一下, 开头作者说的, 这个问题在Windows下有点复杂指定是什么.

这里需要了解一下, Windows和Linux这两大主流操作系统下, 多进程的实现差异

Windows和Linux的多进程差异

Linux下面的multiprocessing基于fork, fork之后所有的本地变量都复制一份, 因此可以使用任意的全局变量;

在Windows下面, 多进程是通过启动新进程完成的, 所有的全局变量都是重新初始化的, 在运行过程中动态生成, 修改过的全局变量是不能使用的.

python: windows和linux下multiprocessing模块创建进程的区别 - 昱成 - 博客园

This does not interoperate very well with Loguru, causing handlers to be added twice without any synchronization or, on the contrary, not being added at all (depending on add() and remove() being called inside or outside the __main__ branch)

简单理解, 可以认为是初始化操作被多次执行导致的问题.

这里来看看具体的示例, Windows的多进程对于代码的影响.

2.4 FastAPI

来进一步扩展一下, 稍作修改, 改掉main.py, 在fastapi中使用

from fastapi import FastAPI
from test_module import Test
from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.test = Test()
    yield

app = FastAPI(lifespan=lifespan)


uvicorn main:app --workers 4 --host 0.0.0.0 --port 8000

这里的多进程是"全局"状态的, 不是由代码本身发起的, 而是运行环境的uvicorn决定的, 这种进程间可以认为是完全"隔离"(进程间通信更为麻烦), 不管是Windows还是Linux代码实例都需要反复初始化.

From the list of deployment concepts from above, using workers would mainly help with the replication part, and a little bit with the restarts, but you still need to take care of the others:

相当于"复刻"一份.

这个机制, 意味着一些需要全局单一实例的, 如数据库, 机器学习模型等的使用将面临很多问题, 如, 多次初始化, 加载多个模型, 这意味将消耗大量的时间和占用极大的资源, 甚至导致其他更严重的问题, 不仅如此, 同时对于如何组织管理代码也是一个挑战.

三. 总结

python的多进程在Linux和Windows上差异很大, 其他语言也有这样的问题吗, 比如Java, go.

doubao:

不同编程语言在多进程 / 多线程的跨平台（如 Linux 和 Windows）表现上，差异程度确实与语言的设计理念、运行时（Runtime）实现以及对系统底层接口的抽象程度密切相关。Python 的多进程在 Linux 和 Windows 上差异显著，主要源于其进程创建机制的底层差异（Linux 使用fork，Windows 没有fork而使用spawn），以及全局状态（如模块导入、变量初始化）在进程复制时的处理方式不同。

其他语言如 Java、Go 的跨平台差异则明显更小，这与它们的设计和运行时实现直接相关，下面具体分析：

1. Java：跨平台一致性较强

Java 的设计目标之一是 “一次编写，到处运行”（Write Once, Run Anywhere），其多线程 / 多进程机制通过 JVM（Java 虚拟机）对系统底层接口进行了高度抽象，因此跨平台差异较小。

  • 多线程：Java 的Thread本质上是对系统线程的封装（Linux 下映射为pthread，Windows 下映射为系统原生线程），但 JVM 统一了线程的创建、调度、同步（如synchronized、Lock）等接口，开发者无需关心底层系统差异。
  • 多进程：Java 通过ProcessBuilder或Runtime.exec()创建子进程，这两种方式在 Linux 和 Windows 上均通过调用系统的进程创建接口（Linux 的fork+exec，Windows 的CreateProcess）实现，但 JVM 屏蔽了底层细节，暴露给开发者的 API 完全一致。
  • 差异点：极端场景下仍可能存在细微差异，例如：
    • 进程间通信（IPC）的底层实现（如管道、共享内存）可能依赖系统特性，但 Java 提供的Socket、FileChannel等跨平台 IPC 方式可规避差异。
    • 信号处理（如SIGINT）在 Linux 和 Windows 上语义不同，但 Java 的ShutdownHook等机制提供了统一的进程退出处理方式。

2. Go：通过 Runtime 层抹平大部分差异

Go 的并发模型（goroutine + channel）是语言级特性，其多进程 / 多线程的跨平台一致性设计更为彻底。

  • goroutine：Go 的轻量级线程（goroutine）由 Go Runtime 而非操作系统直接调度，Runtime 会根据系统核心数将 goroutine 映射到操作系统线程（M:N 调度）。这种机制在 Linux 和 Windows 上的逻辑完全一致，开发者无需感知底层线程实现差异。
  • 多进程：Go 通过os/exec包创建子进程，底层同样调用系统接口（Linux 的fork+exec，Windows 的CreateProcess），但os/exec包对参数（如路径格式、环境变量）进行了跨平台适配（例如自动处理 Windows 的\路径分隔符）。
  • 差异点：主要体现在系统特定的 API 调用上，例如：
    • 对进程的精细控制（如prctl在 Linux 的特殊用法）在 Windows 上可能需要不同实现，但 Go 的标准库已尽量统一常用操作。
    • 信号处理（如SIGTERM vs CTRL_C_EVENT）仍有系统差异，但os/signal包提供了统一的信号监听接口。
